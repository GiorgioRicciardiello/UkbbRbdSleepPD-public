the# Methods: Cross-Sectional Machine Learning Classification of Parkinson's Disease Risk from Actigraphy-Derived RBD Scores

## 1. Study Design and Cohort

We adopted a cross-sectional binary classification framework applied to participants in the UK Biobank (UKBB) who underwent wrist actigraphy recording at the imaging visit (instance 2, approximately 2013–2016). The longitudinal survival structure was collapsed to a binary outcome: subjects who received a confirmed Parkinson's disease (PD) diagnosis at any point during follow-up were labelled as cases (positive class, `y = 1`); all remaining subjects were labelled as controls (`y = 0`).

**Inclusion criteria.** All UKBB actigraphy participants satisfying the following eligibility criteria were included: absence of a neurological exclusion flag at baseline (`neuro_exclude = 0`), acceptable actigraphy recording quality (`acc_bad_quality ≠ True`), and no active night-shift work at the time of recording (`shift_any_i2_p3426 ≠ 1`). The final analytic cohort comprised approximately 580,000 subjects, of whom approximately 3,877 (~0.67%) carried an ever-PD diagnosis.

**Prevalent versus incident design.** Because this analysis treats PD cross-sectionally (ever vs never), prevalent and incident PD cases contribute identically to the outcome label. The distinction is preserved as a covariate in sensitivity analyses but is not a criterion for inclusion/exclusion here. The companion epidemiological temporality analysis (Section 9) restricts to incident cases only.

---

## 2. Outcome Definition

The primary outcome was `outcome_1a_pd_only` (Boolean): ever diagnosed with PD (ICD-10 G20 or algorithmically defined) at any point from the start of UK Biobank follow-up to the last available health-record linkage date. This label is independent of the actigraphy recording date; a subject is a case if they carry a PD diagnosis in the linked hospital episode statistics (HES) or first-occurrence fields, regardless of when the diagnosis occurred relative to the imaging visit.

---

## 3. Feature Set

Features were selected a priori to represent four biologically motivated domains:

### 3.1 Demographics

Age at UKBB recruitment (`cov_age_recruitment_21022`, continuous, years), sex (`cov_sex_31`, binary 0/1), and body-mass index at the imaging visit (`bmi_21001_i2`, continuous, kg/m²). BMI was recorded at instance 2 to minimise temporal distance from the actigraphy recording.

### 3.2 Actigraphy-Derived RBD Probability

The mean nightly RBD-behaviour-disorder (RBD) probability score (`abk_rbd_score_mean`) was generated by a previously validated machine-learning model applied to raw accelerometry continuous wavelet features. This scalar summarises the likelihood that a subject's overnight actigraphy pattern is consistent with RBD across all recorded nights. Higher values indicate greater RBD-like sleep behaviour; the score is continuous and approximately normally distributed.

### 3.3 Polygenic Risk

Two polygenic risk scores were included: `prs_score_pd` (PRS for PD, constructed from GWAS summary statistics) and `prs_score_rbd` (PRS for RBD). PRS scores were standardised to zero mean and unit variance in the genetic analysis subset. Both PRS were available only for subjects of European ancestry (N ≈ 442,000 with non-null PRS).

### 3.4 Ancestry Principal Components

The first ten ancestry principal components from genotyping (`prs_pc1` through `prs_pc10`) were included to control for residual population stratification that may confound PRS–outcome associations.

### 3.5 Time-to-Event Temporal Feature

A time-to-event feature (`time_to_event_log`) was engineered to probe whether the temporal distance between actigraphy recording and PD diagnosis contributes information beyond cross-sectional risk. For confirmed cases, this is defined as:

```
time_to_event_log = log(1 + max(0, days_from_wear_start_to_pd_diagnosis))
```

Prevalent cases (PD diagnosed before wear-time start, yielding negative days) are clipped to zero before log-transformation. For controls, the feature is undefined (censored); it is handled via the fold-specific imputation procedure described in Section 5.3. Two analytical variants were evaluated (Section 5.3).

### 3.6 Excluded Features

All prodromal marker columns (constipation, orthostatic hypotension, depression, erectile dysfunction, anosmia/hyposmia, dream enactment, anxiety) were excluded. This exclusion is intentional: (1) prodromal features are explored separately in the Cox-regression pipeline; (2) inclusion here would conflate the RBD-centric classification objective with a broad prodromal syndrome score.

---

## 4. Class Imbalance

At 0.67% prevalence, the positive class is heavily under-represented (approximately 1 case per 149 controls). All models addressed this imbalance via class reweighting rather than synthetic oversampling (SMOTE was not used):

- **scikit-learn models** (logistic regression, elastic-net, SVM, random forest): a per-sample weight array was computed as:
  ```
  w_i = (n_negative / n_positive)  if y_i = 1
  w_i = 1.0                         if y_i = 0
  ```
  This weight vector was computed on the training fold and passed to `estimator.fit(..., sample_weight=w)`. Random forest additionally used `class_weight='balanced_subsample'` to apply independent reweighting within each bootstrap subsample.

- **XGBoost**: the `scale_pos_weight = n_negative / n_positive` parameter was computed from the training fold and passed at model construction. This internally scales the gradient contributions of positive-class samples, achieving equivalent reweighting within the XGBoost objective.

The reweighting ratio was recomputed independently in each outer training fold to prevent indirect leakage of global class distribution information.

---

## 5. Training Procedure

### 5.1 Nested Stratified Cross-Validation

Model evaluation followed a nested stratified k-fold scheme to yield unbiased generalisation estimates while simultaneously enabling data-driven hyperparameter optimisation:

- **Outer loop**: 5-fold stratified cross-validation (`StratifiedKFold(n_splits=5, shuffle=True, random_state=42)`). Each outer fold provides a held-out test set that contributes one row to the per-fold metrics table.
- **Inner loop**: 3-fold stratified cross-validation (`StratifiedKFold(n_splits=3, shuffle=True, random_state=43)`) used exclusively as the objective surface for Optuna hyperparameter optimisation. The outer test fold is not visible during inner-loop optimisation.

Stratification ensures that the positive-class prevalence in each fold closely matches the full-dataset prevalence, which is critical at 0.67%.

### 5.2 Leakage-Safe Imputation

An `ImputerPipeline` object was instantiated and fit **strictly on the outer training fold** for each of the five outer folds. Fitting on the full dataset before splitting would constitute leakage of distributional information (medians, modes, case percentile values) from held-out samples into training.

- **Continuous features**: median imputation (median computed on training fold).
- **Categorical features**: mode imputation (mode computed on training fold).
- **`time_to_event_log` (controls only)**: see Section 5.3.

The fitted `ImputerPipeline` was then applied `.transform()` to both the training and test portions of each outer fold. The same procedure was repeated independently inside the inner loop for each Optuna trial, using only the inner training fold.

### 5.3 Time-to-Event Imputation Strategies

Two analytical variants were pre-specified and run in parallel to quantify the contribution of the temporal feature:

**`exclude`**: The `time_to_event_log` column is dropped from the feature matrix before training. This provides a clean baseline relying exclusively on cross-sectional RBD scores, demographics, and genetics — no temporal information.

**`jittered`** (recommended for reporting): The column is retained. For controls (NaN entries), the imputer draws each control's fill value independently from a uniform distribution over the interval [Q₃, max] of the case distribution **in the training fold**:

```
fill_control_i ~ Uniform(Q75(cases_train), max(cases_train))
```

The seed is `random_state + fold_index` to ensure reproducibility while varying the draw across folds. This strategy treats controls as "very late converters or non-converters", keeping fill values within the observed case range (no inflation beyond the case maximum) while avoiding the degenerate point-mass artifact. A previously evaluated `constant` strategy (filling all controls with the 95th percentile of training-fold cases) was discarded because it creates a single imputed value shared across all controls, which tree-based models trivially exploit as an implicit class label, inflating AUC by 0.15–0.20 PR-AUC on balanced subsets.

The interpretive consequence is that PR-AUC in the `jittered` condition is higher than in `exclude` for tree models. The difference `PR-AUC(jittered) − PR-AUC(exclude)` quantifies the net temporal signal extractable with a principled imputation, and is used as evidence for (or against) meaningful temporal variation in the feature.

### 5.4 Hyperparameter Optimisation

Optuna (v4.5.0) with the Tree-structured Parzen Estimator (TPE) sampler (`seed=42`) was used for inner-loop hyperparameter search. For each outer fold, 50 Optuna trials were evaluated. The objective function was the **mean average precision score** (PR-AUC) averaged across the three inner folds. Direction was `maximize`.

**Rationale for PR-AUC as the optimisation objective.** At 0.67% prevalence, ROC-AUC can exceed 0.75 while a classifier ranks nearly all positives below the median score, because the vast negative pool dominates the AUC integral. Average precision weights the precision–recall trade-off and is sensitive to the rank ordering of the rare positive class specifically. This makes it the appropriate objective for severely imbalanced binary classification.

**Hyperparameter search spaces** (per model):

| Model | Hyperparameter | Search range |
|---|---|---|
| Logistic | `C` | [1e-3, 1e2] log-uniform |
| Elastic-net | `C` | [1e-3, 1e2] log-uniform |
| | `l1_ratio` | [0.0, 1.0] uniform |
| XGBoost | `max_depth` | {3, …, 8} integer |
| | `learning_rate` | [1e-3, 0.3] log-uniform |
| | `n_estimators` | {100, …, 800} integer |
| | `min_child_weight` | [0.5, 10] log-uniform |
| | `subsample` | [0.5, 1.0] uniform |
| | `colsample_bytree` | [0.5, 1.0] uniform |
| | `reg_lambda` | [1e-3, 10] log-uniform |
| | `reg_alpha` | [1e-4, 10] log-uniform |
| Random Forest | `n_estimators` | {200, …, 800} integer |
| | `max_depth` | {4, …, 20} integer |

| | `max_features` | {sqrt, log2} categorical |

The best hyperparameter set from the inner loop was used to refit a final model on the full outer training fold, which was then evaluated on the held-out outer test fold.

## 6. Model Architectures

### 6.1 Logistic Regression

L2-penalised logistic regression (`sklearn.linear_model.LogisticRegression`, solver `lbfgs`, `max_iter=2000`, `random_state=42`). Features were standardised to zero mean and unit variance using a `StandardScaler` fit on the outer training fold. The regularisation strength `C` was optimised via Optuna.

### 6.2 Elastic-Net Logistic Regression

Logistic regression with elastic-net penalty (`saga` solver, `max_iter=4000`, `random_state=42`). The elastic-net penalty combines L1 and L2 regularisation: `penalty = (1 − l1_ratio) × L2 + l1_ratio × L1`. Features were standardised identically to the logistic model. Both `C` and `l1_ratio` were optimised.

### 6.3 XGBoost

Gradient-boosted trees (`xgboost.XGBClassifier`, `objective='binary:logistic'`, `eval_metric='aucpr'`, `tree_method='hist'`, `n_jobs=1`, `verbosity=0`). The `scale_pos_weight` parameter was set to `n_negative / n_positive` from the training fold. No feature scaling was applied (trees are scale-invariant).

### 6.4 Random Forest

Bootstrap aggregation of decision trees (`sklearn.ensemble.RandomForestClassifier`, `class_weight='balanced_subsample'`, `random_state=42`, `n_jobs=-1`). `balanced_subsample` applies independent class reweighting to each bootstrap replicate, which is more robust than global `balanced` for highly imbalanced data.

### 6.5 Support Vector Machine (SVM)

An RBF-kernel SVM (`sklearn.svm.SVC`, `kernel='rbf'`, `class_weight='balanced'`, `probability=True`, `cache_size=500 MB`, `random_state=42`) was included for completeness. Probability calibration used Platt scaling. Features were standardised with `StandardScaler`. The SVM was excluded from full-cohort runs due to O(n²)–O(n³) training complexity: at n ≈ 580,000 subjects it is computationally intractable and produced inferior discrimination relative to tree models in all subsample evaluations. Results are reported for smoke-test subsets only.

## 7. Evaluation Metrics

For each outer fold, the following metrics were computed from the held-out test predictions:

- **AUC-PR** (average precision score): primary metric for all comparisons.
- **AUC-ROC**: reported alongside AUC-PR for comparability with the broader literature.
- **Classification threshold**: selected per fold via Youden's J statistic, `J = sensitivity + specificity − 1`, evaluated on the outer test fold. The threshold maximising J was applied to convert probabilities to binary predictions.
- **F1 score** (binary, positive class).
- **Accuracy**.
- **Sensitivity** (true positive rate).
- **Specificity** (true negative rate).
- **Positive predictive value (PPV)** and **negative predictive value (NPV)**.
- **Brier score**: mean squared error between predicted probabilities and binary labels; lower is better.
- **Confusion matrix**: TP, FP, TN, FN counts at the Youden threshold.

All per-fold metrics were aggregated as mean ± standard deviation across the five outer folds. Results are reported as `mean ± SD` throughout.

## 8. Feature Explainability

### 8.1 SHAP Values

SHapley Additive exPlanations (SHAP; Lundberg & Lee, 2017) were computed for the final model refit on the full dataset, evaluated on the last outer fold's held-out test set.

**Explainer selection by model class:**

| Model | SHAP explainer | Notes |
|---|---|---|
| XGBoost | `TreeExplainer` | Exact; O(TLD) where T = trees, L = leaves, D = depth |
| Random Forest | `TreeExplainer` | Returns (n, p, 2) in SHAP >= 0.5; positive class extracted as `sv[:, :, 1]` |
| Logistic regression | `LinearExplainer` | Background distribution = training-fold scaled X |
| Elastic-net | `LinearExplainer` | Idem |
| SVM | `KernelExplainer` | Subsampled: 100 background points, max 500 evaluation points, 100 SHAP samples per point |

Raw SHAP values (n × p array, positive class) are saved as `shap_values.npy`; the corresponding feature matrix is saved as `shap_X_eval.csv`. The `available` flag in `shap_summary.csv` records whether the explainer succeeded.

### 8.2 Bootstrap Confidence Intervals on Mean |SHAP|

The standard SHAP summary statistic — mean absolute SHAP value per feature — is a point estimate that does not quantify uncertainty across the held-out sample. We computed bootstrap 95% confidence intervals using 1,000 bootstrap resamples (with replacement) of the rows of the SHAP matrix:

```
Let A = |shap_values|  (n x p matrix)
For b = 1, ..., 1000:
    Draw bootstrap indices idx_b ~ Uniform({1,...,n}) with replacement, size n
    boot_b[j] = mean(A[idx_b, j])  for each feature j
CI_lower[j] = percentile_2.5(boot_b[:, j])
CI_upper[j] = percentile_97.5(boot_b[:, j])
```

All bootstrap procedures used `numpy.random.default_rng(seed=42)`. Results are visualised as horizontal bar charts of mean |SHAP| with error bars representing the 95% CI, sorted by descending mean importance.

### 8.3 SHAP Interaction Plots

For each model, two SHAP dependence scatter plots were generated to examine feature interactions:

1. **RBD score × PRS for PD** (`abk_rbd_score_mean` × `prs_score_pd`): x-axis = raw RBD score, y-axis = SHAP value for RBD score, colour = PRS-PD value.
2. **RBD score × age** (`abk_rbd_score_mean` × `cov_age_recruitment_21022`): x-axis = raw RBD score, y-axis = SHAP value for RBD score, colour = age at recruitment.

Vertical spread at a given x-axis value reflects the modulating influence of the interaction feature on the RBD feature's marginal contribution to the model output. These plots probe whether the predictive value of high RBD scores is amplified in subjects with higher genetic PD risk or older age, providing an interaction signal beyond the main-effect SHAP magnitude.

### 8.4 Permutation Importance

As a model-agnostic complement to SHAP, permutation importance was computed on the last outer fold's test set using `sklearn.inspection.permutation_importance` (`n_repeats=10, random_state=42`). The scoring function was average precision (PR-AUC) to remain consistent with the optimisation objective. Feature importance is reported as mean ± SD change in PR-AUC upon permutation.

## 9. Epidemiological Temporality Analysis

The temporality analysis was conducted outside the ML pipeline to formally test whether higher RBD severity is associated with shorter time to PD onset — a necessary criterion for causal inference under the Bradford Hill framework.

### 9.1 Cohort Restriction

Prevalent PD cases (PD diagnosed prior to wear-time start) were excluded. Temporality requires the exposure (RBD score at wear-time) to precede the outcome (PD diagnosis). The event indicator was `outcome_1a_pd_only__incident` (n = 2,948 incident PD events among 579,749 eligible subjects after exclusion of prevalent cases).

### 9.2 RBD Severity Groups

Subjects were classified into three ordered groups using precomputed percentile thresholds derived from the RBD score distribution on the validation split:

- **Low**: RBD score < p90 (threshold ≈ 2.148)
- **Intermediate**: p90 <= RBD score < p99 (threshold ≈ 5.660)
- **High**: RBD score >= p99

### 9.3 Survival Analysis

Time-to-event was defined as days from the UKBB baseline assessment to the first PD diagnosis (`outcome_1a_pd_only__surv_days`), converted to years. Non-PD subjects were right-censored at last health-record linkage.

**Kaplan-Meier curves** were estimated per group with 95% pointwise confidence intervals. Overall and pairwise log-rank tests were computed using `lifelines.statistics.multivariate_logrank_test` and `logrank_test`.

### 9.4 Cox Proportional Hazards — Categorical Model

A Cox PH model was fit with binary dummy variables for Intermediate (vs Low, reference) and High (vs Low, reference) RBD groups, adjusted for age at recruitment and sex. A ridge penaliser (`penalizer = 0.1`) was applied to stabilise the partial likelihood surface. Hazard ratios (HR) with 95% confidence intervals and Wald p-values are reported.

### 9.5 Cox PH — Ordinal Trend Test

To formally test for a monotonic dose-response relationship, RBD group was recoded as an ordinal integer (Low = 0, Intermediate = 1, High = 2) and entered as a single continuous covariate in a Cox model adjusted for age and sex (`penalizer = 0.1`). The HR represents the multiplicative change in hazard per one-category increase. The Wald p-value for the ordinal coefficient constitutes the trend test (`p_trend`).

### 9.6 Restricted Mean Survival Time (RMST)

The RMST at a horizon of 10 years was computed per group using `lifelines.utils.restricted_mean_survival_time`, providing an absolute measure of PD-free survival that is interpretable without the proportional hazards assumption.

### 9.7 Case-Only Timing Analysis

Among incident PD cases, the time from wear-time start to PD diagnosis was compared across RBD groups using the Kruskal-Wallis H test (non-parametric, no ordering assumption) and the Jonckheere-Terpstra trend test (ordered alternative: decreasing time with increasing RBD severity). If `scipy.stats.jonckheere_terpstra` was unavailable, the Spearman correlation between group rank and time-to-PD was reported as a fallback.

All analyses were implemented in Python (lifelines 0.30.0, scipy 1.16.3). Figures were generated with matplotlib; all output saved to `results/rbd_severity_temporality/`.

## 10. Reproducibility

All random processes were seeded with `random_state = 42` unless otherwise specified:

| Process | Seed |
|---|---|
| Outer CV splits | `StratifiedKFold(..., random_state=42)` |
| Inner CV splits | `StratifiedKFold(..., random_state=43)` (offset to ensure independence) |
| Optuna TPE sampler | `seed=42` |
| ImputerPipeline jittered fill | `numpy.random.default_rng(random_state + fold_index)` — seeded per fold |
| Bootstrap CIs | `numpy.random.default_rng(42)` |
| Permutation importance | `random_state=42` |
| All tree model estimators | `random_state=42` |

The fold-level seed offset for jittered imputation (`random_state + fold_index`) ensures reproducibility across runs while allowing fold-specific variation in control fill values, preventing all five folds from sharing an identical imputation draw.

## 12. Limitations

1. **Cross-sectional collapse of a censored outcome.** Converting a survival outcome to a binary label ignores differential follow-up time. Subjects with shorter observation windows who did not develop PD are labelled as controls even if they would convert given longer follow-up. This introduces outcome misclassification that is non-differential with respect to RBD score (follow-up length is not correlated with RBD score at baseline), biasing classification performance estimates towards the null.

2. **Time-to-event feature for controls.** Any value assigned to controls for `time_to_event_log` is an encoding choice, not observed data. The jittered imputation is principled but introduces stochasticity; the `exclude` variant is the conservative alternative. Readers should weight the `exclude` results as the primary analysis and `jittered` as a sensitivity run probing the temporal signal.

3. **PRS availability.** PRS scores are available only for individuals of European ancestry (~442,000/580,000 subjects). Missing PRS values are imputed to the training-fold median within each outer fold; the fold-level missingness pattern (~24.5% in balanced subsets) may introduce residual confounding in subjects with missing genetics.

4. **Temporal confounding in SHAP interactions.** SHAP interaction plots pool across all held-out subjects. In the `jittered` condition, the apparent RBD × age interaction may be driven in part by the imputed time-to-event values for controls rather than true biological interaction. Interaction plots in the `exclude` condition are unaffected by imputation.

5. **Proportional hazards assumption.** Cox PH models in the temporality analysis assume that hazard ratios are constant over follow-up time. This assumption was not formally tested (e.g., via Schoenfeld residuals) in the primary analysis. Restricted mean survival time (Section 9.6) is provided as a non-parametric complement.

## 13. Pipeline Orchestration and Parallelisation

### 13.1 Serial Execution

The baseline execution mode processes one feature set at a time (`pipeline.py`):

1. Load data for feature set (via `dataset.py`)
2. Build feature matrix X and outcome vector y (via `features.py`, `feature_sets.py`)
3. Run nested CV: for each of 5 outer folds:
   - Split into train/test (stratified, 80/20)
   - For each of 3 inner folds: run Optuna hyperparameter search with PR-AUC objective
   - Refit best model on full outer training set
   - Evaluate on outer test fold; record metrics
4. Save all artifacts to `results/ml_cross_sectional/{feature_set}/{model}_{YYYYMMDD_HHMMSS_UUID}/`
5. After all models: generate per-feature-set ROC/CM report and cross-model comparison figures

**Runtime**: ~2–3 hours for 4 feature sets on 16-core machine (sequential model execution).

### 13.2 Parallel Execution

The parallel mode (`parallel_pipeline.py`) uses `ProcessPoolExecutor` with coarse-grained parallelism:

1. Spawn `n_workers` independent worker processes (one per feature set)
2. Each worker:
   - Receives pickled DataFrame (~20 MB, unpickle overhead ~1–2 seconds)
   - Calls `_run_fs_worker()` which executes `pipeline.run_all()` for that feature set
   - Saves results to timestamped model directories
3. Master process collects results from workers as they complete (`as_completed()`)
4. After all workers finish: master calls `generate_all_cross_fs_plots()` and `run_final_report()` with a shared `run_id`

**Architecture**:
- Context: `mp.get_context("spawn")` for Windows compatibility
- Initialiser: `_init_worker()` sets `OMP_NUM_THREADS=1`, `MKL_NUM_THREADS=1` to prevent BLAS thread oversubscription
- Feature sets load in arbitrary order; jobs are collected via `as_completed()` (unordered completion)

**Expected speedup**: 3–4x on 4 feature sets with 16 cores (coarse-grained parallelism scales well; Amdahl's law overhead minimal for ~100k-sample datasets and internal joblib parallelism per fold).

### 13.3 Run ID Scheme and Traceability

All model runs are assigned a unique timestamped identifier to prevent overwrites across multiple pipeline executions:

**Format**: `{model}_{YYYYMMDD_HHMMSS_UUID}`

where:
- `{model}` = model class name (e.g., `xgboost`, `logistic`, `elasticnet`)
- `YYYYMMDD_HHMMSS` = timestamp (day, hour, minute, second) when the model run started
- `_UUID` = 5-character random alphanumeric suffix (collision prevention)

**Example**: `xgboost_20260418_121649_k9m5p` = XGBoost run on 2026-04-18 at 12:16:49 UTC, collision ID `k9m5p`

**Implication**: Different models within a single feature set may be saved at different times (due to staggered execution or parallelism), so the set of run IDs across models is typically heterogeneous. The final report identifies the most recent run ID and uses that for the report's directory name.

### 14.3 Final Report

The final report is a unified cross-feature-set summary, generated by `results_collector/runner.py:run_final_report()`:

**Function signature**:
```python
def run_final_report(
    results_root: Path,
    feature_sets: tuple[str, ...],
    selection_metric: str = "auc_roc",  # or "auc_pr", "f1"
    include_pr_curve: bool = False,
    report_timestamp: str | None = None,
) -> Path
```

**Workflow**:

1. **Load best models**: For each feature set, load all model run directories and select the best per feature set using `selection_metric` (default: highest AUC-ROC; alternatives: AUC-PR, F1 score).

2. **Identify common run ID**: Extract the run ID timestamp from each loaded model's directory. If multiple distinct run IDs exist (e.g., different feature sets run on different dates), select the most recent timestamp and use it as the report's identifier. This ensures the report directory name reflects which models went into the comparison.

3. **Generate output directory**: Create `results/ml_cross_sectional/_final_report_{run_id}/`

4. **Generate three outputs**:

   a. **`figure_feature_set_comparison.png`** (via `plot_feature_set_comparison()`):
      - Left panel: Overlaid ROC curves for each feature set's best model (one colour per feature set)
      - Optional: Precision-recall curves (if `include_pr_curve=True`)
      - Right panel: Confusion matrices arranged as grid (up to 2 per row, ordered by feature set)
      - Title, legend, metrics displayed per confusion matrix cell

   b. **`figure_feature_set_supplemental.png`** (via `plot_supplemental_figure()`):
      - 2 × 2 grid:
        - `[0, 0]`: Grouped bar chart of key metrics (sensitivity, specificity, F1, accuracy) × feature set
        - `[0, 1]`: SHAP feature importance (top-5 per feature set, stacked vertically); fallback to permutation importance if SHAP unavailable
        - `[1, 0]`: Calibration plot (Brier score vs feature set, with error bars for ±SD)
        - `[1, 1]`: Cohort composition (horizontal stacked bar: # cases, # controls)

   c. **`table_best_model_summary.csv`** (via `make_best_model_summary_table()`):
      - One row per feature set
      - Columns: Feature Set, Model, AUC-ROC ± SD, AUC-PR ± SD, Sensitivity (%), Specificity (%), PPV (%), F1 (%), Accuracy (%), Brier ± SD, TP, FP, TN, FN, N, Threshold*
      - Threshold* = Youden-optimised classification threshold from outer test folds

   d. **`table_all_models_summary.csv`** (via `make_all_models_summary_table()`):
      - One row per (feature_set, model) combination (e.g., 4 feature sets × 4–5 models = 16–20 rows)
      - Same columns as best model summary, allowing direct comparison of all models
